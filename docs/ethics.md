# Ethics and Safety Principles

SYMBORG touches sensitive territory: attention, memory, personal context, communication, and possible future neurotechnology. That requires a conservative design posture.

## Core Principles

### 1. Agency First

The user must remain the speaker, thinker, and decision maker. SYMBORG can suggest, compress, and retrieve, but it should not impersonate or override the user.

### 2. Silence Is a Feature

A cognitive assistant that always speaks is not intelligent. The system must be able to stay silent when the user is overloaded, the context is sensitive, or confidence is low.

### 3. Local-First Memory

Personal memory should stay local by default. Cloud sync, model training, or third-party processing should require explicit user choice and clear explanation.

### 4. No Covert Persuasion

SYMBORG should not manipulate beliefs, emotions, purchases, or political choices through hidden personalization.

### 5. Confidence Must Be Visible

When factual claims matter, the system should show uncertainty and source confidence. It should not produce fluent cues that hide weak evidence.

### 6. Right to Disconnect

The user must be able to pause, mute, delete memory, and use the world without augmentation.

### 7. Right to Remain Unaugmented

SYMBORG should not create a social expectation that every person must use cognitive augmentation to remain employable, competent, or socially competitive.

### 8. Assistive Benefit Before Enhancement Hype

The most responsible high-impact path is assistive communication and accessibility. Consumer enhancement claims should remain modest until measured.

## Red Lines

The project should avoid:

- "Mind reading" claims
- Medical treatment claims
- Covert emotional influence
- Advertising based on cognitive or neural state
- Unconsented recording
- Private memory extraction
- Output that speaks as the user without confirmation
- Social scoring or behavioral ranking

## Evaluation Questions

Before adding a feature, ask:

1. Does this increase user agency or reduce it?
2. Can the user understand what the system knows?
3. Can the user stop it quickly?
4. Would the feature still be acceptable if used in a workplace or school?
5. Does it protect the vulnerable user, or optimize the system owner?
6. Is silence safer than output here?
